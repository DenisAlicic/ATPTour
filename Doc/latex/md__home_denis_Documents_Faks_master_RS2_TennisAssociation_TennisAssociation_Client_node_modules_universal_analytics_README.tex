A node module for Google\textquotesingle{}s \href{http://support.google.com/analytics/bin/answer.py?hl=en&hlrm=de&answer=2790010}{\texttt{ Universal Analytics}} tracking via the \href{https://developers.google.com/analytics/devguides/collection/protocol/v1/}{\texttt{ Measurement Protocol}}.

This module allows tracking data (or rather, users) from within a Node.\+js application. Tracking is initiated on the server side and, if required, does not require any more tracking in the browser.

{\ttfamily universal-\/analytics} is installed and included like any other node module\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\$ npm install universal-\/analytics}

\end{DoxyCode}



\begin{DoxyCode}{0}
\DoxyCodeLine{var ua = require('universal-\/analytics');}
\DoxyCodeLine{}
\DoxyCodeLine{// Or with ES6 import}
\DoxyCodeLine{import ua from 'universal-\/analytics'}

\end{DoxyCode}


Initialization expects at least your Google Analytics account ID\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{var visitor = ua('UA-\/XXXX-\/XX');}

\end{DoxyCode}


This will create a {\ttfamily universal-\/analytics} Visitor instance that you can use and keep around to track a specific client (Not to be confused with the Google Analytics User ID, see \href{\#setting-persistent-parameters}{\texttt{ Setting persistent parameters}} for more information on that). Since no client ID was specified in the constructor\textquotesingle{}s arguments, a random UUID is generated. In case you have a client ID at hand, you can use that to create the visitor\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{var visitor = ua('UA-\/XXXX-\/XX', '6a14abda-\/6b12-\/4578-\/bf66-\/43c754eaeda9');}

\end{DoxyCode}


Starting with Universal Analytics, a UUID v4 is the preferred client ID format. It is therefor necessary to provide a UUID of such type to {\ttfamily universal-\/analytics}. However you can force custom client ID, passing {\ttfamily strict\+Cid\+Format\+: false} in the options\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{var visitor = ua('UA-\/XXXX-\/XX', 'CUSTOM\_CLIENTID\_1', \{strictCidFormat: false\});}

\end{DoxyCode}


If you want to use Google Analytics in http protocol, just include it in the options {\ttfamily http\+: true}, by default will use https\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{var visitor = ua('UA-\/XXXX-\/XX', \{http: true\});}

\end{DoxyCode}


If you want to set User Id you can add it into options\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{var visitor = ua('UA-\/XXXX-\/XX', \{uid: 'as8eknlll'\});}

\end{DoxyCode}
 \href{https://developers.google.com/analytics/devguides/collection/protocol/v1/parameters\#uid}{\texttt{ see about User Id}}

Tracking a pageview without much else is now very simple\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{visitor.pageview("{}/"{}).send()}

\end{DoxyCode}


The first argument for the pageview method is the path of the page to be tracked. Simply calling {\ttfamily pageview()} will not initiate a tracking request. In order to send off tracking to the Google Analytics servers you have two options\+:


\begin{DoxyEnumerate}
\item You can append a {\ttfamily send()} call after {\ttfamily pageview()}. The tracking request is sent asynchronously. This means you will not receive any confirmation when and if it was successful.
\item You can provide a callback function to {\ttfamily pageview()} as an additional, last argument. This callback will be invoked once the tracking request has finished. Any error that occured during the request will be provided to said callback. In that case {\ttfamily send()} is no longer necessary.
\end{DoxyEnumerate}

An example of the callback approach\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{var visitor = ua('UA-\/XXXX-\/XX');}
\DoxyCodeLine{visitor.pageview("{}/"{}, function (err) \{}
\DoxyCodeLine{  // Handle the error if necessary.}
\DoxyCodeLine{  // In case no error is provided you can be sure}
\DoxyCodeLine{  // the request was successfully sent off to Google.}
\DoxyCodeLine{\});}

\end{DoxyCode}

The first argument for the pageview tracking call is the page path. Furthermore, pageview tracking can be improved with the additional parameters to provide the page\textquotesingle{}s hostname and title to Google Analytics. The parameters are provided as arguments after the page path.


\begin{DoxyCode}{0}
\DoxyCodeLine{visitor.pageview("{}/"{}, "{}http://peaksandpies.com"{}, "{}Welcome"{}).send();}

\end{DoxyCode}

\DoxyCodeLine{  // â€¦}

Depending on how you integrate tracking into your app, you might be more comfortable with providing all the tracking data via a params object to the {\ttfamily pageview()} method\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{visitor.pageview(\{dp: "{}/"{}, dt: "{}Welcome"{}, dh: "{}http://peaksandpies.com"{}\}).send();}

\end{DoxyCode}


This code has the exact same effect as the one above. {\ttfamily dp}, {\ttfamily dt}, and {\ttfamily dh} (as in \textquotesingle{}document path\textquotesingle{}, \textquotesingle{}document title\textquotesingle{} and \textquotesingle{}document hostname\textquotesingle{}) are the attribute names used by the Measurement Protocol.

It\textquotesingle{}s mandatory to specify either the page path ({\ttfamily dp}) or document location ({\ttfamily dl}). Google Analytics can not track a pageview without a path. To avoid such erroneous requests, {\ttfamily universal-\/analytics} will deny {\ttfamily pageview()} tracking if the required parameters are omitted.

E-\/commerce tracking in general is a bit more complex. It requires a combination of one call to the {\ttfamily transaction()} method and one or more calls to the {\ttfamily item()} method.


\begin{DoxyCode}{0}
\DoxyCodeLine{visitor}
\DoxyCodeLine{  .transaction("{}trans-\/12345"{}, 500)   // Create transaction trans-\/12345 worth 500 total.}
\DoxyCodeLine{  .item(300, 1, "{}item-\/54321"{})        // Add 1 unit the item item-\/54321 worth 300.}
\DoxyCodeLine{  .item(200, 2, "{}item-\/41325"{})        // Add 2 units the item item-\/41325 worth 200.}
\DoxyCodeLine{  .send()}

\end{DoxyCode}


Once again, daisy-\/chaining simplifies associating the items with the transaction. Officially, nothing but the transaction ID is a requirement for both the transaction and the items. However, providing a minimum set of information (revenue for the transaction, price, quantity and ID for the items) is recommended.

It is also possible to provide the params as an object to both methods\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{visitor}
\DoxyCodeLine{  .transaction(\{ti: "{}trans-\/12345"{}, tr: 500, ts: 50, tt: 100, ta: "{}Partner 13"{}\})}
\DoxyCodeLine{  .item(\{ip: 300, iq: 1, ic: "{}item-\/54321"{}, in: "{}Item 54321"{}, iv: "{}Blue"{}\})}
\DoxyCodeLine{  .item(\{ip: 200, iq: 2, ic: "{}item-\/41325"{}, in: "{}Item 41325"{}, iv: "{}XXL"{}\})}
\DoxyCodeLine{  .send()}

\end{DoxyCode}


In case an additional item has to be added later on or daisy-\/chaining is not available for another reason, each item can be given an associated transaction ID via the params object as well\+:

visitor.\+item(\{ip\+: 100, iq\+: 1, ic\+: \char`\"{}item-\/41325\char`\"{}, in\+: \char`\"{}\+Item 41325\char`\"{}, iv\+: \char`\"{}\+XL\char`\"{}, ti\+: \char`\"{}trans-\/12345\char`\"{}\}).send()

We have seen basic daisy-\/chaining above when calling {\ttfamily send()} right after {\ttfamily pageview()} and {\ttfamily event()}\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{visitor.pageview("{}/"{}).send()}

\end{DoxyCode}


Every call of a tracking method returns a visitor instance you can re-\/use\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{visitor.pageview("{}/"{}).pageview("{}/contact"{}).send()}

\end{DoxyCode}


Granted, the chance of this example actually happening in practice might be rather low.

However, {\ttfamily universal-\/analytics} is smart when it comes to daisy-\/chaining certain calls. In many cases, a {\ttfamily pageview()} call is instantly followed by an {\ttfamily event()} call to track some additional information about the current page. {\ttfamily universal-\/analytics} makes creating the connection between the two easy\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{visitor.pageview("{}/landing-\/page-\/1"{}).event("{}Testing"{}, "{}Button color"{}, "{}Blue"{}).send()}

\end{DoxyCode}
 This is the same as two distinct tracking calls.


\begin{DoxyCode}{0}
\DoxyCodeLine{visitor.pageview("{}/landing-\/page-\/1"{}).send()}
\DoxyCodeLine{visitor.event("{}Testing"{}, "{}Button color"{}, "{}Blue"{}, \{p: "{}/landing-\/page-\/1"{}\}).send()}

\end{DoxyCode}


Daisy-\/chaining is context-\/aware and in this case placing the {\ttfamily event()} call right after the {\ttfamily pageview()} call results in the event being associated with the page path tracking in the {\ttfamily pageview()} call. Even though the attributes ({\ttfamily dp} and {\ttfamily p}) are different internally.

It also works when using a callback since the {\ttfamily this} inside the callback will be the {\ttfamily universal-\/analytics} Visitor instance\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{visitor.pageview("{}/landing-\/page-\/1"{}, function (err) \{}
\DoxyCodeLine{  if (!err) \{}
\DoxyCodeLine{    this.event("{}Testing"{}, "{}Button color"{}, "{}Blue"{}).send()}
\DoxyCodeLine{  \}}
\DoxyCodeLine{\});}

\end{DoxyCode}


More generally, the daisy-\/chaining context keeps all parameters from the previous call around. This means in a situation where similar tracking calls are necessary tracking is simplified\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{visitor}
\DoxyCodeLine{  .event(\{ec: "{}Mail Server"{}, ea: "{}New Team Member Notification sent"{}\})}
\DoxyCodeLine{  .event(\{ea: "{}Invitation sent"{}\})}
\DoxyCodeLine{  .send();}

\end{DoxyCode}


In this example the event category (\char`\"{}\+Mail Server\char`\"{}) is not repeated in the second tracking call.\hypertarget{md__home_denis_Documents_Faks_master_RS2_TennisAssociation_TennisAssociation_Client_node_modules_universal_analytics_README_autotoc_md2910}{}\doxysection{Setting persistent parameters}\label{md__home_denis_Documents_Faks_master_RS2_TennisAssociation_TennisAssociation_Client_node_modules_universal_analytics_README_autotoc_md2910}
Some parameters should be in every tracking call, such as a user ID or custom dimensions that never or hardly change. For such situations a {\ttfamily \#set(key, value)} method is available


\begin{DoxyCode}{0}
\DoxyCodeLine{visitor.set("{}uid"{}, "{}123456789"{});}

\end{DoxyCode}


The uid parameter will be part of every tracking request of that visitor from now on.

For custom dimensions, you will not pass {\ttfamily dimension\#} rather {\ttfamily cd\#}\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{visitor.set("{}cd[1-\/20]"{}, "{}123456789"{}); // [1-\/20] will be the dimension number}

\end{DoxyCode}
\hypertarget{md__home_denis_Documents_Faks_master_RS2_TennisAssociation_TennisAssociation_Client_node_modules_universal_analytics_README_autotoc_md2911}{}\doxysection{Filter application tracking data}\label{md__home_denis_Documents_Faks_master_RS2_TennisAssociation_TennisAssociation_Client_node_modules_universal_analytics_README_autotoc_md2911}
Set a persistent parameter for \href{https://developers.google.com/analytics/devguides/collection/protocol/v1/parameters\#ds}{\texttt{ {\ttfamily Data Source}}} to {\ttfamily app} in order to mark tracking data as {\ttfamily Application}.


\begin{DoxyCode}{0}
\DoxyCodeLine{visitor.set("{}ds"{}, "{}app"{}); // Allows filtering by the 'Application?' field in GA}

\end{DoxyCode}


Then create a new view in Google Analytics of type \textquotesingle{}Application\textquotesingle{}. You will then need to filter the data for that view by creating a new filter that either includes or excludes {\ttfamily Application? Yes} (depending on if you want to show(includes) or hide(excludes) application analytics in a given view).

\hypertarget{md__home_denis_Documents_Faks_master_RS2_TennisAssociation_TennisAssociation_Client_node_modules_universal_analytics_README_autotoc_md2912}{}\doxysection{Session-\/based identification}\label{md__home_denis_Documents_Faks_master_RS2_TennisAssociation_TennisAssociation_Client_node_modules_universal_analytics_README_autotoc_md2912}
In order to make session-\/based apps easier to work with, {\ttfamily universal-\/analytics} also provides a middleware that works in an Expressjs-\/style fashion. It will try to detect a client ID based on the {\ttfamily \+\_\+ga} cookie used by the analytics.\+js client-\/side tracking. Additionally it will store the detected client ID in the current session to recognize the visitor later.


\begin{DoxyCode}{0}
\DoxyCodeLine{var ua = require("{}universal-\/analytics"{});}
\DoxyCodeLine{var express = require("{}express"{});}
\DoxyCodeLine{}
\DoxyCodeLine{var app = express()}
\DoxyCodeLine{}
\DoxyCodeLine{express.use(ua.middleware("{}UA-\/XXXX-\/Y"{}, \{cookieName: '\_ga'\}));}

\end{DoxyCode}


The middleware will attach the {\ttfamily universal analytics} visitor instance to every request ({\ttfamily req.\+visitor}).

Additionally, the module also exposes a {\ttfamily create\+From\+Session} method to create a visitor instance simply based on a session, which is helpful when working with Socket.\+io, etc. where the middleware is not used.


\begin{DoxyCode}{0}
\DoxyCodeLine{var visitor = ua.createFromSession(socket.handshake.session);}

\end{DoxyCode}
